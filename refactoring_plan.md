# `svg.user.js` 重构方案

## 1. 引言

本方案旨在对 `qq5855144/greasyfork` 仓库中的 `svg.user.js` 用户脚本进行全面重构。当前脚本功能强大，但随着功能的不断增加，代码结构变得复杂，维护成本提高。本次重构将重点关注代码的模块化、UI 组件化、图标管理现代化以及整体代码的可读性和可维护性。

## 2. 现有代码分析

`svg.user.js` 是一个 Tampermonkey 用户脚本，主要功能是网页图片采集、预览和批量下载。其主要特点和存在的问题如下：

*   **单一文件结构**: 所有代码（配置、样式、逻辑、UI 构建）均集中在一个 `svg.user.js` 文件中，导致文件庞大，难以阅读和维护。
*   **UI 构建方式**: UI 元素通过字符串拼接 HTML 和直接 DOM 操作创建，缺乏组件化和复用性，修改 UI 布局或样式时容易出错。
*   **样式管理**: 样式通过 `GM_addStyle` 注入，但与 HTML 结构紧密耦合，且缺乏主题管理机制。
*   **图标管理**: SVG 图标直接嵌入在 `DOMBuilder` 模块的 HTML 字符串中，不便于统一管理和替换。
*   **模块化程度**: 尽管代码中存在一些模块（如 `Draggable`, `DynamicListener`, `Notification` 等），但它们之间仍存在较强的耦合，且命名方式有待规范。
*   **代码现代化**: 部分 JavaScript 模式可以更新为更现代的 ES6+ 语法。

## 3. 重构目标

*   **提高可维护性**: 通过模块化和组件化，降低代码耦合度，使代码结构更清晰，易于理解和修改。
*   **增强可扩展性**: 方便未来添加新功能或修改现有功能，例如支持更多图片格式、新的 UI 交互等。
*   **优化 UI 开发**: 引入组件化思想，使 UI 开发更高效、更具可复用性。
*   **现代化代码**: 采用最新的 JavaScript 特性，提升代码质量和执行效率。
*   **统一图标管理**: 建立一套集中管理和渲染图标的机制。

## 4. 技术栈选择与重构策略

考虑到用户脚本的特殊性（需要在浏览器环境中运行，且不宜引入大型框架增加脚本体积），我们将选择轻量级且兼容性良好的技术栈和策略。

### 4.1 UI 框架/组件系统

*   **策略**: 采用**纯 JavaScript 的组件化模式**。不引入额外的 UI 库（如 React, Vue, LitElement），以保持脚本的轻量级和对 Tampermonkey 环境的良好兼容性。我们将设计一个简单的组件基类或工厂函数，用于封装组件的模板、样式和行为。
*   **原因**: 避免因引入第三方库而增加脚本体积和潜在的兼容性问题。通过原生 JS 实现组件化，可以更好地控制脚本的性能和资源占用。

### 4.2 CSS 样式管理

*   **策略**: 保持通过 `GM_addStyle` 注入样式的方式。将所有样式定义集中到一个或多个 CSS 文件中，并在构建时将其打包到脚本中。考虑使用 **CSS 变量**来管理主题颜色、字体大小等，以提高样式的可配置性。
*   **原因**: `GM_addStyle` 是 Tampermonkey 脚本的标准样式注入方式。集中管理 CSS 有助于提高样式的一致性和可维护性。

### 4.3 图标管理

*   **策略**: 将所有 SVG 图标定义提取到一个独立的 `icons.js` 模块中。每个图标定义为一个 JavaScript 字符串常量。提供一个统一的 `createIcon(iconName)` 函数，根据名称返回对应的 SVG 元素或字符串。
*   **原因**: 集中管理图标便于查找、修改和扩展。避免 SVG 代码散落在各处，提高代码的整洁性。

### 4.4 模块化

*   **策略**: 按照功能职责将 `svg.user.js` 拆分为多个独立的 JavaScript 模块。每个模块负责特定的功能，并通过 ES Module 语法进行导入导出。在 Tampermonkey 环境中，可以通过构建工具（如 Webpack 或 Rollup）将这些模块打包成一个最终的用户脚本文件。
*   **原因**: 降低模块间的耦合度，提高代码的可读性、可测试性和可维护性。

### 4.5 代码现代化

*   **策略**: 尽可能使用 ES6+ 语法，如 `const`/`let`, 箭头函数, 模板字符串, 解构赋值, `async/await` 等。遵循 JavaScript 社区的最佳实践。
*   **原因**: 提升代码质量，减少潜在错误，提高开发效率。

## 5. 架构设计

### 5.1 模块划分

新的文件结构将如下所示：

```
/greasyfork/
├── src/
│   ├── index.js             # 脚本入口，负责初始化和协调
│   ├── config.js            # 全局配置项
│   ├── styles.js            # 所有 CSS 样式定义
│   ├── utils/               # 通用工具函数 (debounce, normalizeUrl, simpleHash等)
│   │   ├── index.js
│   │   └── ...
│   ├── components/          # UI 组件
│   │   ├── Component.js     # 组件基类或工厂函数
│   │   ├── FabButton.js     # 浮动按钮组件
│   │   ├── MainModal.js     # 主模态框组件
│   │   ├── PreviewModal.js  # 预览模态框组件
│   │   └── ...
│   ├── icons.js             # SVG 图标定义和渲染函数
│   ├── modules/             # 核心业务逻辑模块
│   │   ├── ImageCollector.js
│   │   ├── Downloader.js
│   │   ├── Deduplication.js
│   │   ├── DynamicListener.js
│   │   └── ...
│   ├── services/            # 外部服务交互模块
│   │   ├── BlobManager.js
│   │   ├── Notification.js
│   │   └── ...
│   └── app.js               # 主应用逻辑，协调模块和组件
└── svg.user.js              # 最终打包的用户脚本文件 (由构建工具生成)
```

### 5.2 模块职责

*   **`index.js`**: 脚本的入口点，负责加载所有模块，并调用 `app.init()` 启动应用。
*   **`config.js`**: 集中管理所有可配置的常量，如按钮大小、Z-index、颜色等。
*   **`styles.js`**: 包含所有 CSS 字符串，通过 `GM_addStyle` 注入。可以根据主题或组件进行组织。
*   **`utils/`**: 存放各种通用工具函数，如节流、防抖、URL 处理、哈希计算等。
*   **`components/`**: 每个文件定义一个独立的 UI 组件。组件应封装自己的 HTML 结构、样式和行为。组件之间通过事件或属性进行通信。
    *   `Component.js`: 提供一个基础的组件创建机制，例如接受一个配置对象，包含 `template` (HTML 字符串或模板函数), `styles` (CSS 字符串), `methods` (组件方法), `events` (事件绑定) 等。
    *   `FabButton.js`: 负责浮动按钮的创建、拖拽逻辑和点击事件。
    *   `MainModal.js`: 负责主图片列表模态框的创建、显示/隐藏、搜索、排序、筛选和批量操作。
    *   `PreviewModal.js`: 负责图片预览模态框的创建、显示/隐藏、图片切换、缩放和旋转。
*   **`icons.js`**: 包含所有 SVG 图标的字符串定义，并提供一个函数来根据名称获取 SVG 元素。
*   **`modules/`**: 存放核心业务逻辑。
    *   `ImageCollector.js`: 负责从页面中收集图片资源，包括 `<img>` 标签、CSS 背景图、动态加载的图片等。
    *   `Downloader.js`: 负责图片的下载，包括单张下载和批量打包下载（ZIP）。
    *   `Deduplication.js`: 负责图片的去重逻辑，包括 URL 去重和内容签名去重。
    *   `DynamicListener.js`: 负责监听页面的滚动和点击事件，触发图片重新扫描。
    *   `Draggable.js`: 负责 UI 元素的拖拽功能。
*   **`services/`**: 存放与外部环境或浏览器 API 交互的模块。
    *   `BlobManager.js`: 管理 Blob URL 的创建和清理。
    *   `Notification.js`: 负责在页面上显示临时通知。
*   **`app.js`**: 作为主应用程序协调器，负责初始化所有模块和组件，绑定全局事件，并管理应用层面的状态和流程。

### 5.3 数据流与状态管理

*   **单向数据流**: 核心数据（如 `globalImageItems`）由 `app.js` 或专门的数据管理模块维护。UI 组件通过属性接收数据，并通过事件向上通知状态变化。
*   **简单状态管理**: 对于 UI 状态，例如模态框的显示/隐藏、筛选条件等，可以通过 `app.js` 中的状态变量进行管理，并通过方法传递给子组件进行渲染。

## 6. 开发流程

1.  **环境搭建**: 配置构建工具（如 Webpack 或 Rollup）以支持 ES Module 导入导出，并将多个 JS 文件打包成一个用户脚本文件。
2.  **模块拆分**: 逐步将 `svg.user.js` 中的代码拆分到上述规划的模块中。
3.  **UI 组件化**: 按照组件化思想重写 UI 的创建和管理逻辑。
4.  **图标重构**: 将 SVG 图标提取到 `icons.js` 模块。
5.  **功能测试**: 对每个重构后的模块和组件进行单元测试和集成测试，确保功能完整性和稳定性。
6.  **性能优化**: 检查重构后的脚本性能，确保不会引入额外的性能开销。
7.  **代码审查**: 进行代码审查，确保代码质量和规范。

## 7. 风险与挑战

*   **Tampermonkey 环境兼容性**: 确保重构后的代码在 Tampermonkey 环境中能够正常运行，特别是构建工具生成的代码。
*   **性能影响**: 模块化和组件化可能会略微增加脚本体积，需要权衡性能和可维护性。
*   **现有功能回归**: 确保所有现有功能在重构后都能正常工作，不引入新的 bug。

## 8. 总结

本次重构将显著提升 `svg.user.js` 的代码质量和可维护性，为未来的功能扩展打下坚实基础。通过模块化、组件化和现代化技术，我们将构建一个更健壮、更易于开发的图片采集工具。
